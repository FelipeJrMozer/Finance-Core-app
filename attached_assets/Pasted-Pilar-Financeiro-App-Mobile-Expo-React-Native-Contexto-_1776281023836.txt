Pilar Financeiro — App Mobile (Expo / React Native)
Contexto do Projeto
Você está desenvolvendo o app mobile companheiro do Pilar Financeiro, uma plataforma brasileira de gestão financeira pessoal. O app web já existe em produção. O app mobile compartilha o mesmo backend e banco de dados — nenhuma nova API precisa ser criada no servidor. Todas as rotas já existem e estão documentadas abaixo.

O repositório do mobile é: https://github.com/FelipeJrMozer/Finance-Core-app

Princípio fundamental: Web ↔ Mobile como espelho
Toda configuração feita no web (tema, categorias, contas, carteiras, notificações) deve refletir imediatamente no mobile e vice-versa.
O estado vem sempre do backend compartilhado — sem estados locais duplicados.
Ao configurar algo no mobile (ex: criar uma transação, mudar tema para escuro), o web deve exibir a mudança imediatamente na próxima sincronização e vice-versa.
Use React Query (@tanstack/react-query) para cache e sincronização, exatamente como o web.

Autenticação
O backend usa sessão via cookie (HTTP-only). Para o mobile:

Endpoint de login: POST /api/auth/login { email, password }
Endpoint de registro: POST /api/auth/register { name, email, password }
Verificar usuário logado: GET /api/auth/user
Logout: POST /api/auth/logout
Todas as requisições devem incluir credentials: "include" (ou configurar cookies via react-native-cookies)
Armazenar sessão com Expo SecureStore

Configuração do cliente HTTP
// lib/api.ts
const BASE_URL = process.env.EXPO_PUBLIC_API_URL;
export async function apiRequest(method: string, path: string, body?: any) {
  const response = await fetch(`${BASE_URL}${path}`, {
    method,
    headers: { "Content-Type": "application/json" },
    credentials: "include",          // cookies de sessão
    body: body ? JSON.stringify(body) : undefined,
  });
  if (!response.ok) throw new Error(await response.text());
  return response.json();
}

Sincronização Web ↔ Mobile
Tema dark/light: PUT /api/settings { theme } → ambos leem GET /api/settings na inicialização
Contas, categorias, transações: criadas em qualquer plataforma aparecem em ambas via React Query + invalidação de cache
Configurações globais: GET /api/settings é a fonte de verdade para todos os toggles de módulos (bills, sinking-funds, pjBusinessType, etc.)
Notificações: GET /api/notifications — mesmo endpoint, lidas em qualquer plataforma marcam como lidas em ambas
Carteiras (wallets): enviar header x-wallet-id em todas as requisições quando wallet ativa ≠ default
Regras de Negócio Importantes
Saldo de contas: sempre usar computedBalance (campo calculado retornado pela API) — NUNCA usar balance (campo stale do banco)
Contas de crédito: type === "credit" têm saldo negativo (fatura a pagar) — tratar como passivo, não como ativo
Transações parceladas: campo installmentNumber / totalInstallments — exibir "2/6" no item
Transações pendentes: isPaid === false — exibir com opacidade reduzida + badge "Pendente"
Valores: sempre Number(value) ao exibir (API retorna strings decimais)
Datas: usar date-fns/locale/pt-BR para formatação
Cartões de crédito: faturas agrupadas por mês/ano via closingDay e dueDay do cartão
Módulos condicionais: verificar GET /api/settings para saber quais módulos estão habilitados por usuário (billsEnabled, sinkingFundsEnabled, etc.)
