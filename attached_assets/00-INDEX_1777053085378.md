# Pacote de correção do app mobile Finance Core

> Análise feita em 24/abr/2026 comparando o app mobile (https://github.com/FelipeJrMozer/Finance-Core-app) com o sistema web atual em produção (`pilar-financeiro.replit.app`).

## Visão geral

O mobile já implementa a maior parte das telas básicas (dashboard, lançamentos, contas, cartões, investimentos, metas, orçamentos, simuladores, PJ, captura automática Android, scanner Pix, AI chat). Aponta para o backend de produção do web e usa os mesmos endpoints `/api/...`. O que falta é **paridade de cálculo, paridade visual com a marca "Pilar Financeiro" e cobertura de funcionalidades avançadas** — sobretudo investimentos profundos, IRPF/DARF, alertas server-side e relatórios.

## Como usar este pacote

1. Cada arquivo `fase-XX-*.md` é um **prompt completo e autônomo** que pode ser colado direto no agente que vai executar a fase.
2. As fases foram ordenadas em **ordem de dependência**: começar pela 1 evita retrabalho nas seguintes.
3. Cada fase tem critérios claros de aceitação (`Definition of Done`) — o agente só fecha a fase se todos forem atendidos.
4. **Nunca pular a Fase 1**: ela conserta cálculos errados que afetam todas as telas posteriores.

## Convenções aplicadas em todas as fases

- **Idioma**: pt-BR em toda copy do app.
- **Moeda**: BRL com `Intl.NumberFormat('pt-BR')`. Sempre tabular nums em valores.
- **Tema**: respeitar `ThemeContext` (já existe). Cores devem usar `colors.primary/danger/success` — nunca hardcoded.
- **Auth**: todo fetch via `apiGet/apiPost/apiPatch/apiDelete/apiFetch` de `services/api.ts` (refresh automático já está pronto).
- **Wallet scope**: endpoints que aceitam `?walletId=` devem ser chamados via `wq(path)` (já implementado em FinanceContext) para isolar workspace.
- **Loading**: usar Skeleton component novo (Fase 9) e nunca deixar tela em branco — sempre `Skeleton` ou `EmptyState` ou dado.
- **Empty state**: sempre via `<EmptyState>` (já existe) com ícone, título, descrição e CTA primário quando aplicável.
- **TestIDs**: toda interação tem `testID` no padrão `{action}-{target}` (ex: `button-add-transaction`, `input-search-transactions`).
- **Dark mode**: tudo deve testar em ambos os modos. Usar `colors` do `ThemeContext`, nunca cores fixas.
- **Erros de rede**: silenciosos no console (`logger.warn`) e visíveis ao usuário via `notifyError`. Nunca crash.
- **Imutabilidade de IDs**: nunca mudar tipo de coluna primária no schema do `lib/db`.
