Atualizações finais (abril/2026) — OBRIGATÓRIAS
O backend passou por code review e mudou algumas regras. Adapte o mobile aos pontos abaixo antes de considerar o trabalho pronto.

A. Registro com consentimento explícito
POST /api/auth/register agora exige os campos termsAccepted: true e privacyAccepted: true no body. Se não vierem, retorna 400.
No app, antes de habilitar o botão "Criar conta", o checkbox de Termos/Privacidade tem que estar marcado e o body deve conter:
{ email, password, firstName, lastName,
  termsAccepted: true, privacyAccepted: true,
  termsVersion, privacyVersion }

termsVersion/privacyVersion vêm de GET /api/legal/versions (faça um useQuery e renderize "Versão X — atualizado em Y" nas telas de Termos e Privacidade e no próprio texto do checkbox). Não hardcode "1.0.0".
B. Sessões e JWT vinculados (sid)
O backend agora embute o sid (id da sessão) dentro do JWT. Quando o usuário encerra uma sessão via DELETE /api/user/sessions/:id, o access token e o refresh token daquela sessão passam a retornar 401 com mensagem "Sessão encerrada".
Tratamento no app:
Se qualquer request retornar 401 com esse message, limpe o storage (SecureStore/AsyncStorage) e redirecione para login sem tentar refresh.
Se receber 401 "Token revogado" ou "Sessão encerrada" no fluxo de refresh, não fique em loop — faça logout imediato.
Ao chamar POST /api/user/sessions/track no boot, envie deviceId estável (use expo-application.applicationId + Device.osInternalBuildId ou equivalente).
C. Reestruturação em 5 planos
Os planos agora são: ESSENCIAL (grátis), PREMIUM, FAMILY, PJ, INVESTIDOR_PRO.

GET /api/subscription/info retorna plan.name como um desses 5 valores (o antigo FREE foi renomeado para ESSENCIAL). Trate legado: se vier FREE, renderize como ESSENCIAL.
Reescreva a tela de Pricing/Assinatura:
Lista vertical com 5 cards (no mobile, scroll vertical) mostrando nome, preço BRL, bullets de benefícios, badge "Mais popular" em PREMIUM.
CTA chama POST /api/stripe/checkout { planName } e abre result.url em WebBrowser.openBrowserAsync (expo-web-browser), aguardando o retorno pelo deep link de sucesso para invalidar ["/api/subscription/info"].
Se o backend retornar 503 com "Price ID não configurado", mostre toast "Plano indisponível no momento" — não quebre a tela.
Crie hooks/useFeatureAccess.ts:
type Feature = "ai" | "investments" | "pj" | "family" | "advancedPortfolio";
export function useFeatureAccess(feature: Feature): boolean

Consumindo GET /api/subscription/info. Use para gate nas telas:
ai → tela de IA / chat.
investments → telas de Investimentos, Dividendos, Performance.
pj → módulo PJ/MEI (DARF, DAS, Livro Caixa).
family → módulo Family.
advancedPortfolio → Correlation, Drawdown, Scorecard, Portfolio Analysis.
Quando o gate bloqueia, mostre tela/card com ícone de cadeado, mensagem "Disponível no plano {X}" e botão "Ver planos" navegando para Pricing.
D. Modo discreto — cobertura total
Além de Dashboard/Contas/Cartões/Transações/Investimentos, aplique maskValue em: Bills, Debts, Goals, Budgets, FinancialReports, Family (saldos do grupo), PJ (receitas/despesas), CardAnalytics. Em gráficos, só mascare tooltips — não os eixos.

E. Split de transação — validação exata
Ao salvar splits (POST /api/transaction-splits), a soma das linhas deve bater com o valor da transação (tolerância R$ 0,01). Mostre a diferença em tempo real no modal e só habilite o botão "Salvar" quando bater.
Ofereça botão "Distribuir igualmente" que reparte o total entre as N linhas (arredondando para 2 casas, sobra na 1ª linha).
F. Anomalias e recap no Dashboard
Card de anomalias deve mostrar a badge colorida por severidade (increasePct >= 100 vermelho, >= 50 âmbar, senão cinza).
Se GET /api/insights/anomalies retornar array vazio, não renderize o card.
Tratamento de erro silencioso: se a query falhar, o dashboard segue sem o card (não quebre o fluxo).
